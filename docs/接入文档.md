# DeskGame Backend 接入文档

本文面向前端、机器人客户端和联调调用方，描述当前后端实际可用的 HTTP 与 WebSocket 接口。

## 1. 基本约定

### 1.1 运行地址

- HTTP 基址：`http://localhost:3000`
- WebSocket 基址：`ws://localhost:3000`

### 1.2 数据约定

- 所有写操作都建议带 `requestId`，用于幂等与日志追踪
- `openId` 作为玩家唯一标识
- `roomId` 作为房间唯一标识
- 失败响应统一为：

```json
{
  "error": "错误信息"
}
```

### 1.3 当前阶段模型

游戏阶段固定按以下顺序推进：

`lobby -> roleSelection -> bet -> action -> settlement -> discussionVote -> review`

游戏状态：

- `wait`
- `selecting`
- `playing`
- `ended`

## 2. HTTP 接口

### 2.1 创建房间

`POST /rooms`

请求体：

```json
{
  "ownerOpenId": "owner-001",
  "roomConfig": {
    "playerCount": 5,
    "roleConfig": "independent"
  },
  "requestId": "req-create-room"
}
```

说明：

- 房主在创建时会自动作为第一个玩家加入房间
- `playerCount` 目前限制为 `5-10`
- `roleConfig` 当前支持 `independent` 和 `faction`

成功响应：

```json
{
  "events": [
    {
      "name": "RoomCreated",
      "roomId": "room-id",
      "roomCode": "123456",
      "ownerOpenId": "owner-001",
      "gameState": "wait",
      "currentRound": 0,
      "currentStage": "lobby",
      "version": 1
    }
  ],
  "room": {
    "roomId": "room-id",
    "roomCode": "123456"
  }
}
```

### 2.2 加入房间

`POST /rooms/:roomId/players`

请求体：

```json
{
  "openId": "player-002",
  "nickname": "Bob",
  "avatar": "https://example.com/avatar.png",
  "requestId": "req-join-room"
}
```

### 2.3 离开房间

`DELETE /rooms/:roomId/players/:openId`

请求体可选：

```json
{
  "requestId": "req-leave-room"
}
```

### 2.4 更新房间配置

`POST /rooms/:roomId/config`

请求体：

```json
{
  "openId": "owner-001",
  "roomConfig": {
    "playerCount": 6,
    "roleConfig": "faction"
  },
  "requestId": "req-update-config"
}
```

说明：

- 仅房主可修改配置
- 一般应在 `lobby` 阶段执行

### 2.5 设置准备状态

`POST /rooms/:roomId/ready`

请求体：

```json
{
  "openId": "player-002",
  "ready": true,
  "requestId": "req-ready"
}
```

说明：

- 所有已加入玩家都准备，且人数满足 `roomConfig.playerCount` 时，房间会自动进入 `roleSelection`

### 2.6 确认身份选择

`POST /rooms/:roomId/role-selection`

请求体：

```json
{
  "openId": "player-002",
  "roleId": "passenger",
  "requestId": "req-role-selection"
}
```

说明：

- `roleId` 必须来自服务端发给该玩家的候选身份列表
- 全员确认后自动进入 `bet`，并初始化环境牌序列

### 2.7 提交押牌

`POST /rooms/:roomId/bets`

押牌动作：

```json
{
  "openId": "player-002",
  "actionCard": "scold",
  "requestId": "req-bet-card"
}
```

不押牌：

```json
{
  "openId": "player-003",
  "passedBet": true,
  "requestId": "req-bet-pass"
}
```

说明：

- `actionCard` 与 `passedBet` 二选一
- 不押牌玩家本回合不能发言、不能投票

### 2.8 提交投票

`POST /rooms/:roomId/votes`

请求体：

```json
{
  "openId": "player-002",
  "voteTarget": "player-004",
  "requestId": "req-vote"
}
```

### 2.9 推进阶段

`POST /rooms/:roomId/stage/advance`

手动推进：

```json
{
  "openId": "owner-001",
  "requestId": "req-advance"
}
```

超时推进：

```json
{
  "timeoutFlag": true,
  "requestId": "req-timeout-advance"
}
```

说明：

- 一般由房主驱动推进
- 某些阶段也可由超时逻辑触发推进

### 2.10 查询房间快照

`GET /rooms/:roomId`

成功响应：

```json
{
  "room": {
    "roomId": "room-id",
    "ownerOpenId": "owner-001",
    "roomCode": "123456",
    "gameState": "playing",
    "currentStage": "bet",
    "currentRound": 1,
    "version": 12
  }
}
```

## 3. 用户接口

### 3.1 创建/更新用户

`POST /users`

请求体：

```json
{
  "openId": "wx-open-id",
  "nickname": "Alice",
  "unionId": "wx-union-id",
  "avatarUrl": "https://example.com/avatar.png",
  "gender": 1,
  "city": "Shanghai",
  "province": "Shanghai",
  "country": "CN",
  "language": "zh_CN"
}
```

说明：

- `openId` 和 `nickname` 为必填字段
- 其余字段可选
- 若该 `openId` 已存在则更新，否则创建（幂等写入）
- `gender`：`0` 未知，`1` 男，`2` 女

成功响应（HTTP 200）：

```json
{
  "user": {
    "openId": "wx-open-id",
    "nickname": "Alice",
    "unionId": "wx-union-id",
    "avatarUrl": "https://example.com/avatar.png",
    "gender": 1,
    "city": "Shanghai",
    "province": "Shanghai",
    "country": "CN",
    "language": "zh_CN",
    "createdAt": "2026-03-28T10:00:00.000Z",
    "updatedAt": "2026-03-28T10:00:00.000Z"
  }
}
```

### 3.2 查询用户

`GET /users/:openId`

成功响应（HTTP 200）：

```json
{
  "user": {
    "openId": "wx-open-id",
    "nickname": "Alice",
    "unionId": "wx-union-id",
    "avatarUrl": "https://example.com/avatar.png",
    "gender": 1,
    "city": "Shanghai",
    "province": "Shanghai",
    "country": "CN",
    "language": "zh_CN",
    "createdAt": "2026-03-28T10:00:00.000Z",
    "updatedAt": "2026-03-28T10:00:00.000Z"
  }
}
```

若用户不存在，返回 HTTP 404：

```json
{ "error": "User wx-open-id not found" }
```

### 3.3 更新用户资料

`PUT /users/:openId`

请求体（所有字段均为可选）：

```json
{
  "nickname": "Alice Updated",
  "avatarUrl": "https://example.com/new-avatar.png",
  "unionId": "wx-union-id",
  "gender": 2,
  "city": "Beijing",
  "province": "Beijing",
  "country": "CN",
  "language": "zh_CN"
}
```

说明：

- 仅更新请求体中出现的字段
- 若用户不存在，返回 HTTP 404

成功响应（HTTP 200）同 3.2 格式。

## 4. WebSocket 接入

### 4.1 连接方式

服务端使用 Socket.IO。

客户端发送的事件名固定为 `command`。
服务端广播的事件名固定为 `event`。
服务端错误事件名固定为 `error`。

### 4.2 消息封装

客户端命令：

```json
{
  "type": "COMMAND",
  "name": "CreateRoom",
  "requestId": "req-001",
  "payload": {
    "ownerOpenId": "owner-001",
    "roomConfig": {
      "playerCount": 5,
      "roleConfig": "independent"
    }
  }
}
```

服务端事件：

```json
{
  "type": "EVENT",
  "name": "RoomCreated",
  "roomId": "room-id",
  "payload": {}
}
```

服务端错误：

```json
{
  "type": "ERROR",
  "name": "CommandError",
  "requestId": "req-001",
  "payload": {
    "code": "COMMAND_ERROR",
    "message": "错误信息"
  }
}
```

### 4.3 支持的命令名

- `CreateRoom`
- `JoinRoom`
- `LeaveRoom`
- `UpdateRoomConfig`
- `SetReady`
- `ConfirmRoleSelection`
- `SubmitBet`
- `SubmitVote`
- `AdvanceStage`

## 5. 推荐联调时序

推荐最小联调顺序：

1. 创建房间
2. 其他玩家加入房间
3. 房主更新配置（可选）
4. 所有玩家设置 ready
5. 服务端广播 `RoleSelectionStarted`
6. 每个玩家提交 `ConfirmRoleSelection`
7. 服务端广播 `RoleSelectionCompleted`
8. 每个玩家提交 `SubmitBet`
9. 房主或超时逻辑调用 `AdvanceStage`
10. 依次接收 `EnvironmentRevealed`、`RoundSettled`、`StageAdvanced`
11. 进入 `discussionVote` 后由有资格玩家提交 `SubmitVote`
12. 房主或超时逻辑再次调用 `AdvanceStage`
13. 接收 `VoteResolved`，若未结束则进入下一回合，否则进入 `review`

## 6. 联调注意事项

- 不要假设客户端可以直接修改阶段，阶段推进只能由服务端命令驱动
- 不要假设所有玩家都能投票，不押牌玩家会失去当回合投票资格
- 角色候选列表是按玩家分发的，客户端必须使用服务端给自己的候选身份
- 同一 `requestId` 应避免重复生成不同语义请求
- 当前实现是内存态，重启服务会丢失房间与对局数据
